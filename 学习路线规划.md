### 3.4 关键代码逐行注解

#### 数据清洗 — 每一步的原因

```python
# ===== 步骤1：查看整体情况 =====
df.info()
# 输出：RangeIndex: 541909 entries（总共54万行）
# 关键看 Dtype（类型是否正确）和 Non-Null Count（哪些列有空值）

# ===== 步骤2：统计空值 =====
df.isnull().sum()
# CustomerID  135080 null  ← 25%的数据没有客户ID！因为很多是匿名/游客购买
# Description  1454 null   ← 商品描述缺失，只占0.2%，可以删

# ===== 步骤3：删除空值 =====
df = df.dropna(subset=['Description'])  # 删1454行，可忽略
df = df.dropna(subset=['CustomerID'])   # 删13.5万行！这个决策很重
# 为什么删而不是填充？CustomerID是客户标识，不是数值，"猜一个"没有意义
# Trade-off：损失25%数据 vs 无法做客户分析。如果目标是客户分析，必须删

# ===== 步骤4：删重复行 =====
df.duplicated().sum()  # 看有多少重复
df = df.drop_duplicates()

# ===== 步骤5：过滤异常数值 =====
(df['Quantity'] <= 0).sum()  # 8872行 ← 退货/冲账
(df['UnitPrice'] <= 0).sum()  # 40行 ← 脏数据/赠品
df = df[(df['Quantity'] > 0) & (df['UnitPrice'] > 0)]
# 为什么不用 df = df[df['Quantity'] > 0] 两行分开写？
# 用 & 一次性过滤更高效，而且逻辑清晰："两个条件同时满足才保留"

# ===== 步骤6：日期类型转换 =====
df['InvoiceDate'] = pd.to_datetime(df['InvoiceDate'])
# 转换前是 object（字符串），不能做时间运算
# 转换后是 datetime64，可以 .dt.year、.dt.month、时间加减
```

#### RFM 打分 — 每一步的原因

```python
# ===== 步骤1：确定分析基准日 =====
# 基准日 = 数据中最后一天 + 1天
# 为什么+1？因为如果客户在最后一天购买，Recency=0天（合理）
# 如果不+1，max日期当天买的客户 Recency=0，逻辑上也没大问题，但工业上习惯+1
analysis_date = df['InvoiceDate'].max() + timedelta(days=1)

# ===== 步骤2：按客户聚合 R/F/M =====
rfm_data = df.groupby('CustomerID').agg({
    'InvoiceDate': lambda x: (analysis_date - x.max()).days,
    # ↑ Recency：距离基准日多少天没买了
    # x.max() 取每个客户最后一次购买日期
    # analysis_date - x.max() 得到 timedelta，.days 转成数字
    # 值越小越好（刚买过）

    'InvoiceNo': 'nunique',
    # ↑ Frequency：有多少笔独立订单
    # 为什么用 nunique 而不是 count？
    # 因为一个订单号下面有多行（多个商品），count会重复计算
    # 值越大越好（经常买）

    'TotalAmount': 'sum'
    # ↑ Monetary：总共花了多少钱
    # 值越大越好（花得多）
})

# ===== 步骤3：用分位数打1-5分 =====
# pd.qcut：按分位数把数据切成q等份
# labels=[1,2,3,4,5]：第1份得1分（最差），第5份得5分（最好）
# duplicates='drop'：如果分位数重复（很多人都是"买了1次"），自动降级处理
rfm_data['F_Score'] = pd.qcut(rfm_data['Frequency'], q=5, labels=[1,2,3,4,5])
rfm_data['M_Score'] = pd.qcut(rfm_data['Monetary'], q=5, labels=[1,2,3,4,5])

# Recency 是反的：天数小=好，天数大=差
# qcut 默认分1=最小=最好，所以 6-score 把方向反过来
# 比如某客户 Recency 是最小的那一档 → qcut=5 → 最终 R_Score = 6-5 = 1 ？？？
# 不，等等：Recency小=好客户，如果他的recency落在最小的那20%
# → qcut labels=1（最小=第1档）→ 但我们想要最高分！
# → 所以 6-1=5分！对的，逻辑正确
rfm_data['R_Score'] = 6 - pd.qcut(rfm_data['Recency'], q=5, labels=[1,2,3,4,5])

# ===== 步骤4：拼三位数 RFM_Score =====
# 555 = R=5 + F=5 + M=5 = 最近买 + 经常买 + 花得多 = 最佳客户
# 111 = 很久没来 + 只买过一次 + 花得少 = 沉睡客户
rfm_data['RFM_Score'] = (rfm_data['R_Score'].astype(str) +
                          rfm_data['F_Score'].astype(str) +
                          rfm_data['M_Score'].astype(str))

# ===== 步骤5：把分数映射到业务策略 =====
def segment_customers(row):
    if row['RFM_Score'] in ['555', '554', '544', '545', '454', '455', '445']:
        return 'Champions'          # 冠军客户 — 奖励他们
    elif row['RFM_Score'] in ['543', '444', '435', '355', '354', '345', '344', '335']:
        return 'Loyal Customers'    # 忠实客户 — 向上销售
    # ... 共10类
    else:
        return 'Hibernating'        # 冬眠客户 — 低价值，低活跃
```

#### 同期群分析（Cohort）— 每一步的原因

```python
# ===== 步骤1：定义每个客户的"同期群" =====
# 同期群 = 同一个月首次购买的客户
df_cohort = df.groupby('CustomerID')['OrderMonth'].min().reset_index()
df_cohort.columns = ['CustomerID', 'CohortMonth']
# 结果：每个客户一个 CohortMonth，比如
# CustomerID=12345 → CohortMonth=2011-01（他第一次买东西是2011年1月）

# ===== 步骤2：把同期群信息合并回主表 =====
df = df.merge(df_cohort, on='CustomerID')
# 现在每行交易多了一个字段 CohortMonth

# ===== 步骤3：计算这个交易是客户"第几个月的交易" =====
df['PeriodNumber'] = (df['OrderMonth'] - df['CohortMonth']).dt.days // 30
# PeriodNumber=0：首月（当月购买）
# PeriodNumber=1：首次购买后的第2个月
# // 30 整数除法，粗糙地把一个月≈30天，学术上可以接受

# ===== 步骤4：透视表 =====
# 行 = CohortMonth（哪个同期群）
# 列 = PeriodNumber（第几个月）
# 值 = 该同期群在该月份有多少人购买了
cohort_data = df.groupby(['CohortMonth', 'PeriodNumber'])['CustomerID'].nunique().unstack()

# ===== 步骤5：留存率 = 每期人数 / 首期人数 =====
# iloc[:, 0] 是 PeriodNumber=0 列，即每个同期群的初始人数
retention = cohort_data.divide(cohort_data.iloc[:, 0], axis=0)
# axis=0 表示"逐行除"，即每行除以该行的第一个值
```

### 4.2 Day 3（约2小时）：数据清洗 + 国家销售分析

#### 分析前思考（写代码前先想清楚）

**数据清洗：**

1. **为什么要删 CustomerID 为空的 13.5 万行？** 数据清洗没有标准答案，取决于分析目标。目标是客户分析（RFM、Cohort），没有 CustomerID 就无法追踪客户行为 → 必须删。如果目标是整体营收趋势 → 可以保留。**先定目标，再定清洗策略。**
2. **Quantity ≤ 0 为什么视为异常？** 负数是退货/冲账，0 是无效行。如果不删，聚合求和时正负抵消，GMV 会失真。判断标准：这些行代表的是"逆向流程"而非"销售行为"，分析销售时排除，分析退货率时反而需要它们。
3. **为什么先 `describe()` 再看分布？** 清洗不是背步骤，而是根据数据实际情况做决策。`describe()` 看 min/max/分位数，`info()` 看类型和空值——这两个是每次拿到新数据的"体检"动作。

**国家销售分析：**

1. **这个分析回答什么问题？** "市场集中度有多高？资源应该投在哪里？" 不是为了画柱状图而画。
2. **为什么用 4 个聚合指标而不是只用营收？** 营收看规模，订单数看交易频率，客户数看市场渗透，三者结合才能判断一个市场是"大而健康"还是"大但依赖少数大客户"。
3. **Top 12 不是 Top 10 也不是 Top 20——数字本身不是重点。** 12 是经验值，实际中根据数据特点调整：国家数少就全画，国家数多就取一个能让柱子清晰可辨的数量。关键是"为什么选这个数字"，而不是记住 12。

### 4.3 Day 4（约2小时）：月度趋势 + 热销产品

#### 分析前思考

**月度趋势：**

1. **为什么看趋势而不是看总量？** 总量告诉你"卖了多少"，趋势告诉你"在变好还是变差"。趋势里藏着的季节性规律（旺季什么时候）、增长拐点（什么时候开始加速/减速），才是做决策的依据。
2. **为什么用面积图而不是裸折线？** 面积图强调"体量感"，适合营收这种累积量；折线图强调"变化率"，适合增长率这种衍生指标。二者强调的叙事不同。选什么图取决于你想让读者第一眼感受到什么。
3. **数据中 2010-12 只有 1 天数据——怎么办？** 月度对比时，不完整的月份会严重拉低均值。你需要在分析中标注这一点，而不是假装数据是完整的。这是分析师的诚实。

**热销产品：**

1. **按营收排还是按销量排？** 按营收排反映"赚钱能力"，按销量排反映"人气"。高价低频商品（如家具）营收高但销量低，低价高频商品（如日用品）销量高但营收低。两种排序揭露不同的事实，你应该都看，然后解释差异。
2. **Top 产品集中度比排名更重要。** 前 5 个产品占总营收的 20% 还是 80%？这决定了是"头部依赖型"还是"长尾分散型"业务。如果是前者，断货一个爆款就是灾难。
3. **产品名可能不一致（同一个东西多个写法）。** 这是真实数据的常见问题——Description 是人工录入的，没有标准化。UK 数据就有这种情况，你可以在分析中标注这个限制。Olist 有品类体系，这个问题会小很多。

### 4.4 Day 5（约2.5小时）：客户消费额分层 + 价格带分析

#### 分析前思考

**客户消费额分层：**

1. **为什么要分层？** 因为"平均"会骗人。一个花 10 万的客户和 999 个花 10 块的客户，平均值是 ~110，完全掩盖了极端分布。分层让你看到客户结构：少数人贡献大部分收入（帕累托效应），还是人人均等？
2. **分层的阈值怎么定？** 不能拍脑袋。先跑 `customer_spending['TotalSpent'].describe([0.25, 0.5, 0.75, 0.9, 0.95, 0.99])`，看分位数的跳跃点。如果 80% 的人花费 < 200，那 200 就是一个天然的断层。**阈值要从数据里找，不是抄来的。** 代码里的 3000/1000/200 只是这个特定数据集的参考值，换一组数据这些数字就没意义了。
3. **分层之后要干什么？** 不是画个饼图就收工。每一层对应不同的运营策略：VIP 要维护（专属客服、提前体验），Occasional 要激活（首单优惠、召回推送）。**没有后续行动的分层只是数字游戏。**
4. **饼图 vs 柱状图的选择。** 饼图适合展示"部分占整体的比例"，但超过 5 个扇区就看不清了；柱状图适合对比绝对值。这里用饼图是因为核心信息是"VIP 贡献了多少%"，但如果要对比各层的客单价（人均消费），柱状图更好。

**价格带分析：**

1. **为什么分析价格带？** 回答"什么价位的产品卖最好"，指导定价和选品策略。如果主力价格带是 £2-5，那进 £50 的货就是自嗨。
2. **价格带的边界怎么定？** 和客户分层一样，不能拍脑袋。看 `df['UnitPrice'].describe()` 的分位数分布。0.5/2/5/10 这些数字是基于数据实际分布的——Medium (£2-5) 是门槛中最宽的区间，说明大部分商品集中在这个价位。
3. **堆叠柱状图能回答什么问题？** "各国的价格带结构一样吗？" 如果 UK 的 Premium 占比 30% 而其他国家的 Premium 占比 <5%，说明 UK 客户的价格接受度更高，可以投更高端的产品线。如果结构一样 → 全球定价策略可以统一。
4. **注意：这里分析的是单品价格，不是客单价。** 一个订单买 10 件 £2 的商品，客单价是 £20，但价格带分析把每件都归入 Medium。两个维度回答不同问题——价格带看单品定价策略，客单价看客户消费能力。

#### 分析前思考

1. **Dashboard 和单图分析的区别是什么？** 单图分析是你自己在探索数据，Dashboard 是你把结论讲给别人听。前者可以散、可以多、可以随意；后者必须精、必须少、必须有逻辑顺序。

### 4.6 Day 7（约2.5小时）：RFM 分析

#### 分析前思考

**这是整个项目最重要的模块。** RFM 不是学三个字母，是学一种"用数据量化客户价值"的思维方式。

**模型层面：**

1. **为什么这三个维度？** R（最近一次购买）= 活跃度，F（购买频率）= 忠诚度，M（消费金额）= 贡献度。三个维度覆盖了客户关系的三个本质问题：他还记得我们吗？他经常来吗？他值得投入吗？缺任何一个都会失准——R 低但 F/M 高的人是被遗忘的高价值客户，你只看金额就漏掉了。
2. **基准日为什么 +1 天？** 不是死规矩，是一种保守处理。如果客户刚好在最后一天买了东西，Recency=0，这在逻辑上合理（就是今天买的）。不 +1 差别不大，但面试官可能会问——你只需要解释清楚你的选择。
3. **R 的打分为什么要反转（6-score）？** qcut 默认 1=最小值=最好，但 Recency 是天数，越小越好。如果用原始分，Recency=1 天的人得 1 分（最差），完全反了。反转的逻辑：把"小=好"映射到"分数高=好"，让 R/F/M 三个分数可以加总对比。

**方法层面：**

1. **为什么用 qcut（分位数分箱）而不是固定阈值？** 固定阈值（如 Recency<30 天=5 分）在数据分布变化时会失效。qcut 保证每档人数相等（各 20%），自适应数据分布。代价是：即使两个客户实际差距很大，在边界附近也可能被分到相邻档。两种方法各有利弊，面试时能说出选择理由比记住用哪个更重要。
2. **qcut 的 `duplicates='drop'` 是干什么的？** 当大量客户有相同的值（比如 F=1，只买过一次），qcut 无法切出等份的五段 → 报错。`duplicates='drop'` 自动减少分箱数，比如原本要 5 档变成 3 档。这是真实数据中几乎必踩的坑。
3. **125 种 RFM Score（5×5×5）为什么要合并成 10 类？** 125 种太细，运营团队不可能对 125 群客户分别制定策略。合并的原则是"相邻分数行为相似"——544 和 545 的客户行为差异远小于 555 和 111。合并后的 10 类每一类对应一个清晰的运营动作。
4. **规则的边界是主观的。** Champions 包含哪些 Score（555/554/544…）是分析师的判断，不同业务可以调整。重点是：你的分类逻辑能不能自圆其说？每一类的定义和对应的运营动作是否一致？

**业务层面：**

1. **RFM 的终点不是分类表，是"对每一类客户做什么"。** Champions → 奖励计划（不是打折，是"限定款抢先体验"这类不伤利润的甜头）。At Risk → 召回邮件 + 限时优惠。Hibernating → 放弃还是激活？要看激活成本和新客获取成本的对比。
2. **RFM 刻画的是历史行为，不是未来价值。** 一个 Hibernating 客户可能是因为产品不满意，也可能是因为搬家了。RFM 告诉你"谁值得关注"，但不能告诉你"为什么流失"。要回答"为什么"，需要结合其他数据（退货记录、客服投诉、评分等）。

**这是重点模块。** RFM 分析包含：R/F/M 计算 → qcut 打分 → 拼 RFM_Score → 规则分类 → 4 子图可视化。

#### 完整代码

```python
# ===== Cell 11：RFM 计算 — 按客户聚合 R/F/M =====

# 步骤1：确定分析基准日
# 基准日 = 数据中最后一天 + 1天
# 为什么 +1？确保最后一天购买的客户 Recency 最小为 0（今天买的），而不是 1
# 不 +1 也行，但面试时能解释清楚你的选择即可
analysis_date = df['InvoiceDate'].max() + timedelta(days=1)
print(f"分析基准日: {analysis_date.date()}")  # 大约 2011-12-10

# 步骤2：按 CustomerID 聚合 R/F/M 三个维度
rfm = df.groupby('CustomerID').agg({
    # Recency（R）：距离基准日有多少天没买了
    # x.max() 取该客户最后一次购买的日期
    # analysis_date - 最后购买日期 = timedelta，.days 转成整数天数
    # 值越小越好（刚买过的人更可能再买）
    'InvoiceDate': lambda x: (analysis_date - x.max()).days,

    # Frequency（F）：有多少笔独立订单
    # nunique（去重计数），不是 count
    # 为什么？一个订单号下面可能有多个商品（多行），count 会重复计算
    # 值越大越好（常买的人是忠实客户）
    'InvoiceNo': 'nunique',

    # Monetary（M）：总共花了多少钱
    # 值越大越好（花得多的人是高价值客户）
    'TotalAmount': 'sum'
}).reset_index()

# 重命名列，让后续代码更可读
rfm.columns = ['CustomerID', 'Recency', 'Frequency', 'Monetary']

print(f"共 {len(rfm)} 个客户参加了 RFM 分析")
print(rfm.describe())  # 先看分布，建立对数据的直觉
```

```python
# ===== Cell 12：qcut 打分 — 把连续值转成 1-5 分 =====

# 步骤1：Frequency 和 Monetary 用 qcut 打分
# pd.qcut：按分位数把数据切成 q 等份，每份人数相等（各 20%）
# labels=[1,2,3,4,5]：最差的那 20% 得 1 分，最好的 20% 得 5 分
# duplicates='drop'：**关键参数**
#   当大量客户有相同值时（比如 F=1，很多人只买了一次），qcut 无法切出等份的 5 段
#   不加这个参数会报错 ValueError: Bin edges must be unique
#   加了之后自动减少分箱数：如果只有 3 个唯一值，就切成 3 档（1,2,3 分）
rfm['F_Score'] = pd.qcut(
    rfm['Frequency'], q=5, labels=[1, 2, 3, 4, 5], duplicates='drop'
)
rfm['M_Score'] = pd.qcut(
    rfm['Monetary'], q=5, labels=[1, 2, 3, 4, 5], duplicates='drop'
)

# 步骤2：Recency 打分 — 需要反转方向
# Recency 是天数，越小 = 越好（刚买过）
# qcut 默认：天数最小的那 20% → label=1（最差），天数最大的那 20% → label=5（最好）
# 这和我们想要的完全反了！
# 反转方法：6 - score
#   天数最小的客户 → qcut 给 1 → 6-1=5 分（最好）✅
#   天数最大的客户 → qcut 给 5 → 6-5=1 分（最差）✅
rfm['R_Score'] = 6 - pd.qcut(
    rfm['Recency'], q=5, labels=[1, 2, 3, 4, 5], duplicates='drop'
)

# 步骤3：确保分数是整数类型（qcut 返回 category 类型）
rfm['R_Score'] = rfm['R_Score'].astype(int)
rfm['F_Score'] = rfm['F_Score'].astype(int)
```

### 4.7 Day 8（约2.5小时）：购物篮分析

#### 分析前思考

1. **这个分析回答什么问题？** "买了 A 的人更可能买 B 吗？" 如果是 → 捆绑销售、推荐位、满减组合。注意：关联 ≠ 因果。买了热狗的人也买了可乐，是因为一起买方便，不是热狗导致了可乐消费。
2. **三个指标分别代表什么？各自有什么局限？**
   - **Support（支持度）= 同时买 A 和 B 的交易 / 总交易数。** 回答"这两个东西一起买的概率多大"。局限：稀有的组合 Support 很低，但可能 Lift 很高——比如买了咖啡机的人总是买专业咖啡豆，但买咖啡豆的总人数很少。
   - **Confidence（置信度）= 买 A 的人中也买了 B 的比例。** 回答"买 A 之后买 B 的可能性多大"。局限：如果 B 本来就是热门商品（比如塑料袋），那 Confidence 天然就高，但这不是 A 的功劳。
   - **Lift（提升度）= Confidence / P(B)。** 回答"A 对 B 的购买有多大的提升效果"。Lift > 1 才说明 A 真的"推动了"B。这是三个指标里最有信息量的一个，因为它排除了 B 本身人气的影响。
   - **只看 Confidence 不看 Lift 是新手最常见的错误。** 你会发现很多规则 Confidence 高达 80% 但 Lift < 1——不是因为 A 推动了 B，而是 B 本来就人人买。
3. **为什么手写而不是调包（mlxtend/apyori）？** 学习阶段手写一遍能理解三个指标的计算逻辑。生产环境直接用 mlxtend 的 `apriori()` 和 `association_rules()`，更快更稳。两件事不冲突——先手写理解原理，以后调包更快。
4. **为什么只保留出现 >=20 次的商品？** 购物篮分析的候选商品对数量是 O(n²)，54 万行数据如果全部两两配对，计算量爆炸。过滤低频商品（没人买的东西不可能有强关联）大幅减少计算量。20 是经验值，你可以试试 10 和 50，看规则数量和质量的变化。

```python
# ===== Cell 16：定义 Support / Confidence / Lift 函数 =====

# 这三个函数是整个购物篮分析的核心，手动实现一遍比调包更有助于理解原理

def calc_support(itemset, transactions):
    """
    计算 Support（支持度）
    = 同时包含 itemset 中所有商品的交易数 / 总交易数
    例：1000 笔订单中，有 50 笔同时买了 A 和 B → Support(A,B) = 50/1000 = 0.05 = 5%
    含义：这两个商品一起出现的概率有多大
    注意：itemset 是一个 set，可以包含 1 个或多个商品
    """
    # (transactions[list(itemset)].sum(axis=1) == len(itemset))
    #   → 对每一行（每个订单），检查该订单是否买了 itemset 中所有商品
    #   → 如果买了全部 len(itemset) 件，该行就是 True
    # .sum() → 统计有多少行是 True
    # / len(transactions) → 除以总订单数 = 比例
    n_transactions = len(transactions)
    matching = (transactions[list(itemset)].sum(axis=1) == len(itemset)).sum()
    return matching / n_transactions


def calc_confidence(antecedent, consequent, transactions):
    """
    计算 Confidence（置信度）
    = 同时包含 A 和 B 的交易数 / 包含 A 的交易数
    = Support(A ∪ B) / Support(A)
    含义：买了 A 的人中，有多大比例也买了 B
    条件概率 P(B|A)
    """
    ant_support = calc_support(antecedent, transactions)
    if ant_support == 0:
        return 0  # A 本身没人买，置信度无意义
    both = antecedent.union(consequent)
    return calc_support(both, transactions) / ant_support


def calc_lift(antecedent, consequent, transactions):
    """
    计算 Lift（提升度）
    = Confidence(A→B) / Support(B)
    = P(B|A) / P(B)
    含义：知道客户买了 A 之后，对"他也会买 B"这件事的预测能力提升了多少倍
    - Lift = 1：买不买 A 对买 B 没影响（A 和 B 独立）
    - Lift > 1：买了 A 的人更可能买 B（正向关联）→ 有推荐价值
    - Lift < 1：买了 A 的人反而不太买 B（负向关联）→ 可能是替代品
    这是三个指标中最重要的一个！它排除了 B 本身人气的影响
    """
    cons_support = calc_support(consequent, transactions)
    if cons_support == 0:
        return 0
    return calc_confidence(antecedent, consequent, transactions) / cons_support
```

```python
    for j, item_b in enumerate(items):
        if i >= j:
            continue  # 跳过 A=A 和重复对（(A,B)只算一次，但两个方向都算规则）

        pair_count += 1
        # 每处理 5000 对打印一次进度（防止以为卡死了）
        if pair_count % 5000 == 0:
            print(f"  已处理 {pair_count} 对...")

        # 计算 A→B
        ante = {item_a}
        cons = {item_b}
        conf_ab = calc_confidence(ante, cons, basket_filtered)
        lift_ab = calc_lift(ante, cons, basket_filtered)

        # 筛选条件：Lift > 1（才有推荐价值），Confidence > 阈值（才有实际意义）
        if lift_ab > 1 and conf_ab > 0.1:  # Confidence > 10% 意味着有实际商业价值

            rules.append({
                'Antecedent': item_a,
                'Consequent': item_b,
                'Support': round(support_ab, 4),
                'Confidence': round(conf_ab, 4),

        lift_ba = calc_lift(cons, ante, basket_filtered)

        if lift_ba > 1 and conf_ba > 0.1:

                'Consequent': item_a,
```

### 4.8 Day 9（约2.5小时）：同期群分析

#### 分析前思考

1. **Cohort 分析回答什么问题？** "不同时间进来的客户，后续表现一样吗？" 如果 2011-01 的新客留存率 45%，2011-09 的只有 20% → 拉新渠道质量在下降，或者 9 月进来的很多是冲着促销来的"薅羊毛"用户。普通月度留存图只看"总活跃用户数"在变不变，Cohort 把原因拆开——是各批新客都在流失，还是某一批拉胯？
2. **为什么用月度而不是按天？** 按天做 Cohort 数据太稀疏（大部分客户不会天天买），热力图几乎全是空白。月度是频率和分析精度之间的平衡点。B2B 业务可能用季度，高频交易（如外卖）可以用周——粒度取决于业务节奏，不是固定公式。
3. **`// 30` 整数除法是近似处理，为什么可以接受？** 严格来说一个月不是 30 天，28/29/30/31 天不等。用 `// 30` 会把边界上的用户归到错误的 PeriodNumber（比如 2 月 28 号买的算 PeriodNumber=0，3 月 1 号买的也算 PeriodNumber=0，但实际只差了 1 天）。对宏观趋势分析来说，这个误差分散在大量用户中，不影响结论。如果要精确到日历月，代码复杂度增加 3 倍，性价比低。
4. **留存率 = 每期人数 / 首期人数，为什么用首期做分母？** 因为 Cohort 分析的核心指标是"首月来的人，有多少第 N 个月还来"。分母统一为首期人数，不同 Cohort 的留存率才有可比性。如果用当期总人数做分母 → 受新客涌入影响，看不出真实留存能力。
5. **留存曲线和热力图分别强调什么？** 热力图看的是结构——哪些 Cohort 的哪个 Period 最弱；留存曲线看的是整体衰减模式——留存是"断崖式下降"（第一月后就没了）还是"渐进式衰减"。两条信息合起来：衰减快不快（曲线），哪个 Cohort 拉低了整体（热力图）。
6. **Cohort 分析的常见陷阱。**
   - **首月数据不完整：** 如果 2011-12 的新客 Cohort 只有 1 天数据（12 月 1 号），那它的 PeriodNumber=0 就是 1 天行为，和其他 Cohort 的整月对比毫无意义。需要标注或排除。
   - **留存率对初始规模敏感：** 只有 3 个新客的 Cohort，1 个人回来就是 33% 留存率——没有统计意义。小 Cohort 的高留存率不要过度解读。
   - **只看留存率不看绝对人数：** 2011-01 拉了 1000 新客，第 3 个月留存 30% = 300 人。2011-09 拉了 3000 新客，第 3 个月留存 20% = 600 人。后者留存率低但留存人数翻倍——你要的是留存率还是留存规模？取决于业务目标。

Cohort 分析步骤：定义 first purchase month → merge → 计算 PeriodNumber → pivot → heatmap → 留存曲线。

```python
cohort_first = df.groupby('CustomerID')['OrderMonth'].min().reset_index()

print("各月份新客数量:")
```

```python
# ===== Cell 20：计算 PeriodNumber — 这个交易是客户第几个月的交易 =====

# 步骤1：把 OrderMonth 和 CohortMonth 都转成 datetime
# to_period('M') 转成月份 Period，方便做减法
df_cohort['OrderMonth_dt'] = pd.to_datetime(df_cohort['OrderMonth'] + '-01')
df_cohort['CohortMonth_dt'] = pd.to_datetime(df_cohort['CohortMonth'] + '-01')

# 步骤2：计算 PeriodNumber
# (当前月份 - 首次购买月份) 的天数差 ÷ 30
# 为什么用 // 30 而不是精确到日历月？
#   精确到日历月需要考虑不同月份天数差异（28/29/30/31），代码复杂度增加 3 倍
#   对宏观趋势分析来说，30 天近似的误差分散在大量用户中，不影响结论
#   学术和工业界都用这个近似方法
df_cohort['PeriodNumber'] = (
    (df_cohort['OrderMonth_dt'] - df_cohort['CohortMonth_dt']).dt.days // 30
)

# 步骤3：验证 PeriodNumber 分布
print("PeriodNumber 分布:")
print(df_cohort['PeriodNumber'].value_counts().sort_index().head(15))
# PeriodNumber=0 应该是最多的（首月所有客户都在）
# PeriodNumber 越大，人数越少（不是所有客户都会持续购买）
```

```python
# ===== Cell 21：透视 — Cohort × Period 留存矩阵 =====

# 步骤1：构建 Cohort 矩阵
# 行 = CohortMonth（哪个同期群），列 = PeriodNumber（第几个月）
# 值 = 该同期群在该月份有多少独立客户购买了
cohort_matrix = df_cohort.groupby(['CohortMonth', 'PeriodNumber'])['CustomerID'].nunique().unstack()

# 步骤2：过滤掉太小的 Cohort（<30 人，没有统计意义）
# 小 Cohort 的留存率波动太大（1个人回来就是很大比例的留存）
min_cohort_size = 30
valid_cohorts = cohort_sizes[cohort_sizes >= min_cohort_size].index
cohort_matrix = cohort_matrix.loc[cohort_matrix.index.isin(valid_cohorts)]

print(f"Cohort 矩阵: {cohort_matrix.shape[0]} 个月 × {cohort_matrix.shape[1]} 期")
print(f"\n剔除的 Cohort（< {min_cohort_size} 人）:")
small_cohorts = cohort_sizes[cohort_sizes < min_cohort_size]
if len(small_cohorts) > 0:
    for month, size in small_cohorts.items():
        print(f"  {month}: {size} 人")

# 步骤3：计算留存率矩阵
# 留存率 = 第 N 期人数 / 第 0 期人数（该 Cohort 的首月人数）
# .divide(series, axis=0) → 每一行除以该行 PeriodNumber=0 的值
cohort_size_p0 = cohort_matrix[0]  # 每期首月人数
retention = cohort_matrix.divide(cohort_size_p0, axis=0)

print(retention.iloc[:5, :6].round(3))
```

### 5.3 Day 10（约3小时）：数据整合架构

#### 分析前思考

**这是整个 Olist 项目最重要的一天。** JOIN 错了，后面所有分析都建立在错误的数据上。

1. **为什么以 order_items 作为事实表起点？** 维度建模的核心原则：事实表存的是"发生了什么事"（一笔订单中的一个商品项），维度表存的是"这个东西是谁/是什么"（客户、产品、卖家）。以 order_items 为起点，向外 JOIN 各维度表，逻辑最清晰。如果以 orders 为起点，后面 JOIN order_items 时行数会膨胀，但那是合理的——订单和商品是 1:N 的自然关系。
2. **payments 为什么必须先聚合再 JOIN？** 一个订单可能有两次支付（信用卡 80% + 优惠券 20%）。如果不聚合，JOIN 后订单行数会翻倍，GMV 也会被重复计算。一个简单的验证：JOIN 后 `df['payment_value'].sum()` 必须等于聚合前 `order_payments['payment_value'].sum()`。
3. **reviews 为什么取 min(score)？** 一个订单可能有多条评分（不同语言）。取平均还是取最低？取最低是保守策略——如果客户在英文评论打了 5 分、葡语评论打了 1 分，min=1 更接近真实体验。取平均会掩盖极端不满。选 max 过于乐观。没有绝对正确的答案，但你需要说清楚为什么这样选。
4. **order_status 只保留 delivered——这是分析范围的重大决定。** 约 97% 的订单是 delivered，3% 被过滤掉。这意味着：取消订单的原因你分析不了，物流时效只计算了已送达的（过滤掉送达失败的偏差），评分也只来自完成交易的用户（沉默的不满者不会出现）。这些限制你需要在报告中坦诚说明，而不是假装不存在。
5. **JOIN 后的数据验证比 JOIN 本身更重要。** 每 JOIN 一步，验证行数是否合理变化、关键数值列的和是否保持一致。如果 `.merge()` 后行数暴增 10 倍 → 大概率是匹配键有重复，需要先 dedup。不要等最后画图时才发现数据不对。

### 5.4 Day 11（约2小时）：数据清洗 + 整体概览

#### 分析前思考

1. **和 UK 最大的不同：数据清洗的重点变了。** UK 数据是单表，"清洗"主要是删空值、去重复、过滤异常。Olist 是 9 表 JOIN 的产物，"清洗"的重点是验证 JOIN 的正确性——行数对不对、金额和不和原始数据一致、有没有因为 JOIN 产生的重复行。
2. **缺失值要先分类再看。** 产品品类缺失 → 产品信息不完整，填 `'unknown'` 并单独统计。配送日期缺失 → 可能是未送达的订单（但你只保留了 delivered，应该不会出现）。评分缺失 → 客户没评价，是分析盲区。**每遇到一个缺失值，先判断它是"不会填"（数据质量问题）还是"不能填"（业务逻辑决定的缺失），处理方法完全不同。**
3. **KPI 概览的价值不在于数字本身，在于建立直觉。** 总 GMV 多少？10 万还是 1000 万？平均客单价多少？配送平均几天？这些数字是你对整个业务的"体感"，后面的分析都会基于这个体感来判断"这个数字正常吗"。
4. **`describe()` 不是走过场。** 重点看：max 是否离谱（price 有 99999 的异常值？），分位数之间的跳跃是否平滑（如果 75% 分位=100，max=50000，说明有极端值），mean 和 median 的差距是否大（大 = 分布偏斜，中位数更有参考价值）。

### 5.5 Day 12（约2小时）：时间趋势分析

#### 分析前思考

1. **GMV 和订单量为什么要放一起看？** 单独看 GMV 上升 → 可能是营收变好，也可能只是涨价了。单独看订单量上升 → 可能是用户变多，也可能是促销拉低客单价。两条曲线放一起，看它们的"分歧点"——订单量涨但 GMV 跌 → 客单价在下降，可能是在用低价引流。订单量跌但 GMV 涨 → 大单客户在增加。**趋势分析的价值在"交叉验证"，单指标容易误判。**
2. **Olist 的时间跨度和 UK 不同。** UK 只有 1 年（2010.12-2011.12），Olist 有近 2 年（2016.09-2018.10）。更长的跨度意味着：能看清年度的季节性规律（巴西的黑五、圣诞节），但也意味着数据早期和晚期可能有结构性变化（平台从小变大，用户结构变了）。如果整体趋势被早期低基数拉得"一片向好"，考虑只看最近 12 个月。
3. **星期/时段分析能回答什么？** "什么时候做促销效果最好？" 如果周日晚上是下单高峰 → 周日晚上推送优惠通知。如果工作日下午是低谷 → 折扣活动安排在这个时段。这个分析比月度趋势更"可操作"，因为它直接指导运营动作的时间点。
4. **Olist 的时间戳是巴西本地时间还是 UTC？** 需要检查。如果是 UTC，巴西跨越多个时区（UTC-2 到 UTC-5），不转换的话时段分析会有偏移。这是真实数据分析中常见的"时区坑"。

### 5.6 Day 13（约2小时）：地理分析

#### 分析前思考

1. **地理分析在 Olist 中比 UK 更有价值。** UK 数据只有一个国家主导（UK 占 82%），地理分析本质是"验证 UK 集中度"。Olist 覆盖巴西 27 个州，更有意义的问题是："销售集中在圣保罗吗？其他州的渗透率如何？" 这直接指导仓储选址和区域营销预算。
2. **只看销售额排名会漏掉什么？** 圣保罗州人口 4600 万（巴西第一），销售额第一几乎是必然的。更有价值的视角是**人均消费**——哪个州的客单价最高？哪个州的复购率最高？这需要把销售额除以该州的客户数，而不是只看绝对 GMV。
3. **经纬度数据（geolocation 表）要不要用？** 100 万行经纬度数据的价值在于点状可视化（城市的销售热力分布），而不是柱状图能替代的。但用 Plotly/Folium 做地图可视化有一定学习成本。建议：Day 13 先做按州的柱状图分析（快速出结论），地图热力图留到 Streamlit 阶段再做（给面试官看）。
4. **卖家和客户的地理分布应该分开分析。** 卖家集中在东南部（SP/MG/RJ）而客户遍布全国 → 物流成本是核心痛点。这个视角 UK 数据完全没有，是 Olist 特有的洞察。

### 5.7 Day 14（约2.5小时）：产品品类分析

#### 分析前思考

1. **品类体系和散装 Description 的根本区别。** UK 数据用 Description 做产品分析就像用收银小票的"商品名"做分类——同一个东西可能有 20 种写法。Olist 有 product_category_name 这个标准化分类（72 个类别），分析的可靠性完全不在一个量级。这是 Olist 相比 UK 的结构性优势。
2. **单维度排名是不够的，要交叉分析。** 只排名品类销售额 → "家居装饰卖最多"（这几乎不需要分析就能猜到）。有价值的是交叉维度：
   - 高销售额 + 低评分 = 在卖垃圾，品控有问题
   - 低销售额 + 高评分 = 潜力品类，可能因为曝光不足
   - 高销售额 + 长配送时间 = 物流是瓶颈，不是需求问题
   - 高销售额 + 低客单价 = 走量型，不适合提价
3. **品类数量 72 个，全画柱状图不行。** Top 10-15 就够了，其余归为 "Others"。但要在注释里标注 Others 的占比——如果 Others 占 40%，说明品类结构分散，头腰部边界不明显，不是"少数品类主导"的模式。
4. **这个分析可以直接产出业务建议。** "A 品类评分最低 → 优先排查品控""B 品类配送比平均慢 5 天 → 检查对应供应商的发货流程""C 品类只有 3 个卖家在卖 → 平台可以主动招商"。品类分析的终点不是排名表，是可以执行的动作清单。

### 5.8 Day 15（约3小时）：RFM 客户分层

#### 分析前思考

1. **customer_id vs customer_unique_id —— 这次的关键决策。** UK 数据一个客户只有一个 CustomerID，Olist 不同——一个自然人（customer_unique_id）可能在平台上有多个 customer_id（换设备、换邮箱、重新注册）。如果你用 customer_id 做 RFM，一个忠实客户会被拆成多个"不同客户"，R/F/M 全部失真。**数据建模第一步：搞清楚每一列的业务含义，尤其是唯一标识。**
2. **和 UK 的 RFM 对比本身就是分析亮点。** 两个数据集都做了 RFM，结论是否相似？都是帕累托效应（少数人贡献多数收入）？如果 UK 的 VIP 占比 2% 贡献 50% 营收，Olist 的 VIP 占比差异很大 → 为什么？是平台类型不同（批发 vs 零售）？还是用户结构差异？**跨数据集的对比让你的分析有深度，而不是"我又做了一遍 RFM"。**
3. **Olist 有更多维度可以验证 RFM 的合理性。** UK 只有交易数据，Olist 还有评分、配送时效。你可以验证：Champions 的评分是否显著高于 Hibernating？Champions 的配送体验是否更好？这不仅是验证，也是在回答"好的客户体验是否带来更高的客户价值"。
4. **RFM 方法论本身不需要重学一遍。** Day 7 的"分析前思考"里关于 qcut、阈值、反转、10 类合并的所有讨论都适用于这里。你应该能独立解释每个决策，而不是再抄一遍代码。列名变了、数据变了，思维框架不变。

#### 这次和 UK 不一样的地方

- Olist 的客户标识是 `customer_unique_id`（一个人可能用多次 `customer_id` 下单）
- 你的分析粒度用 `customer_unique_id`
- 计算 R / F / M 的逻辑完全一样，只是列名不同
- **Olist 独有加分：** 可以按 Segment 分析评分和配送体验差异

```python
# ===== Cell 22：RFM 计算 — Olist 版本 =====

# 步骤1：分析基准日
analysis_date = df['order_purchase_timestamp'].max() + timedelta(days=1)
print(f"Olist 分析基准日: {analysis_date.date()}")

# 步骤2：按 customer_unique_id 聚合 R/F/M
# 关键区别：用 customer_unique_id 而不是 customer_id！
# 同一个自然人可能换设备重新注册 → 多个 customer_id
# 如果用 customer_id → 一个忠实客户被拆成多个"普通客户" → RFM 全错
rfm = df.groupby('customer_unique_id').agg({
    # R：距基准日有多少天没下单了
    'order_purchase_timestamp': lambda x: (analysis_date - x.max()).days,
    # F：下了多少笔订单（nunique 去重）
    'order_id': 'nunique',
    # M：总共花了多少钱（价格+运费）
    'item_total': 'sum'
}).reset_index()

rfm.columns = ['customer_unique_id', 'Recency', 'Frequency', 'Monetary']

# 步骤3：过滤异常值
# Monetary <= 0：退款的 → 删除（如果是全退的话）
print(f"Monetary <= 0 的客户: {(rfm['Monetary'] <= 0).sum()}")
rfm = rfm[rfm['Monetary'] > 0]

print(f"\n共 {len(rfm)} 个客户参加 RFM 分析")
print(rfm.describe())
```

```python
# ===== Cell 23：qcut 打分 + 分段 =====

# Frequency 和 Monetary 打分
rfm['F_Score'] = pd.qcut(
    rfm['Frequency'], q=5, labels=[1, 2, 3, 4, 5], duplicates='drop'
).astype(int)
rfm['M_Score'] = pd.qcut(
    rfm['Monetary'], q=5, labels=[1, 2, 3, 4, 5], duplicates='drop'
).astype(int)

# Recency 反转（天数小=好，但 qcut 默认小=低分 → 用 6-score 反转）
rfm['R_Score'] = 6 - pd.qcut(
    rfm['Recency'], q=5, labels=[1, 2, 3, 4, 5], duplicates='drop'
).astype(int)

# 拼三位数 RFM_Score
rfm['RFM_Score'] = (
    rfm['R_Score'].astype(str) +
    rfm['F_Score'].astype(str) +
    rfm['M_Score'].astype(str)
)

# 分段函数（和 UK 逻辑一致，可以微调阈值）
def segment_olist(row):
    score = row['RFM_Score']
    r, f, m = row['R_Score'], row['F_Score'], row['M_Score']

    if score in ['555', '554', '544', '545', '454', '455', '445']:
        return 'Champions'
    elif score in ['543', '444', '435', '355', '354', '345', '344', '335']:
        return 'Loyal Customers'
    elif score in ['553', '552', '551', '442', '441', '452', '451', '443']:
        return 'Potential Loyalists'
    elif r >= 4 and f <= 2:
        return 'New Customers'
    elif score in ['525', '524', '523', '515', '514', '513']:
        return 'Promising'
    elif score in ['535', '534', '533', '434', '433', '432', '532', '531']:
        return 'Need Attention'
    elif r <= 2 and (f >= 3 or m >= 3):
        return 'At Risk'
    elif r <= 2 and f <= 2 and m <= 2:
        return 'Hibernating'
    elif r >= 3 and f <= 2 and m <= 2:
        return 'About to Sleep'
    else:
        return 'Others'

rfm['Segment'] = rfm.apply(segment_olist, axis=1)

# 分段汇总
seg_summary = rfm.groupby('Segment').agg({
    'customer_unique_id': 'count',
    'Recency': 'mean',
    'Frequency': 'mean',
    'Monetary': ['mean', 'sum']
}).round(1)
seg_summary.columns = ['Count', 'Avg_Recency', 'Avg_Frequency', 'Avg_Monetary', 'Total_Revenue']
seg_summary['Pct_Customers'] = (seg_summary['Count'] / seg_summary['Count'].sum() * 100).round(1)
seg_summary['Pct_Revenue'] = (seg_summary['Total_Revenue'] / seg_summary['Total_Revenue'].sum() * 100).round(1)
print(seg_summary.sort_values('Total_Revenue', ascending=False))
```

### 5.9 Day 16（约2.5小时）：同期群留存分析

#### 分析前思考

1. **Olist 的 Cohort 比 UK 多一个维度：按 state 分组。** 圣保罗州的客户留存率和北部偏远州的客户留存率一样吗？如果不一样，可能是物流体验差异导致的——配送快的地方客户留存高。这在 UK 数据中是无法分析的（因为 UK 只有一个国家，地理差异不大）。**能按维度拆解 Cohort 是 Olist 的独特优势，也是面试时最能展示分析深度的点。**
2. **同一个 client 在同一个月有多笔订单——先去重再算留存。** UK 的粒度是按月统计交易次数，Olist 你需要明确：客户在当月买过就算"留存"，还是必须买满 N 笔才算？前者是"活跃留存"，后者是"高频留存"。选哪个取决于业务关心什么——大多数业务关心前者，因为"还在买"本身就是好信号。
3. **Olist 只有 delivered 订单参与了 Cohort 分析——这是一个需要注意的限制。** 如果客户的订单被取消了（但前面已经发过货了），他在取消月份是否算"活跃"？你只保留了 delivered，所以取消订单的客户在 Cohort 中会表现为"不活跃"。但这个"不活跃"可能不是流失，而是数据范围的限制。**任何数据范围的限制都应该在分析结论中坦白。**
4. **时间跨度 2 年的 Cohort 可以分析"平台成熟度"。** 如果 2016 年的 Cohort 留存率比 2018 年的低 → 平台在改善，早期客户体验不够好。如果趋势相反 → 平台可能在恶化，或者竞争加剧。这是比单纯的"留存率是多少"更高一层的洞察。

```python
# ===== Cell 26：Cohort — merge + PeriodNumber + 去重 =====

# 步骤1：合并 CohortMonth
df_cohort = df.merge(cohort_first, on='customer_unique_id', how='left')

# 步骤2：计算 PeriodNumber
df_cohort['OrderMonth_dt'] = pd.to_datetime(df_cohort['OrderMonth'] + '-01')
df_cohort['CohortMonth_dt'] = pd.to_datetime(df_cohort['CohortMonth'] + '-01')
df_cohort['PeriodNumber'] = (
    (df_cohort['OrderMonth_dt'] - df_cohort['CohortMonth_dt']).dt.days // 30
)

# 步骤3：去重 — 同一客户同一月在同一个 Cohort 只算一次
# 关键区别：UK 数据一个 Customer 每月基本一个订单，Olist 可能有多个
cohort_grouped = df_cohort.groupby(
    ['CohortMonth', 'PeriodNumber', 'customer_unique_id']
).size().reset_index()

# 步骤4：透视 — Cohort × Period 留存矩阵
cohort_matrix = cohort_grouped.groupby(
    ['CohortMonth', 'PeriodNumber']
)['customer_unique_id'].nunique().unstack()

# 过滤小 Cohort（< 30 人）
valid = cohort_sizes[cohort_sizes >= 30].index
cohort_matrix = cohort_matrix.loc[cohort_matrix.index.isin(valid)]

# 留存率 = 每期人数 / 首期人数
retention = cohort_matrix.divide(cohort_matrix[0], axis=0)
retention = retention.iloc[:, :13]  # 前 13 期（一年）

print(f"Cohort 矩阵: {retention.shape[0]} 个月 × {retention.shape[1]} 期")
```

### 5.10 Day 17（约3小时）：Olist 独有分析维度

**这些是 UK 数据没有的，你的核心竞争力所在。**

#### 分析前思考（整体）

这一天包含四个独立分析模块。每个都是小而精的"一个问题 + 一个结论"，而不是长篇大论。核心思路：**交叉分析**——把两个维度的数据放在一起看，找出它们之间的关系。

#### 5.10.1 物流时效分析

**分析前思考：**

1. **物流时效分析的商业价值。** 巴西国土面积世界第五，亚马逊雨林横跨北部，偏远地区的配送挑战非常大。物流效率直接影响客户满意度和复购率——这是 Olist 独特的分析视角，UK 小国家的配送差异不大。
2. **只看平均值会掩盖问题。** 平均配送 8 天——看起来还行。但如果 50% 的订单 5 天到，30% 的订单 15 天到 → 有 30% 的用户体验极差。**看 distribution，不是看 mean。** 直方图是必需品，均值是辅助信息。
3. **超时率的定义要明确。** `delivery_delayed = actual > estimated`。但如果平台给偏远地区的预估本来就长（15 天），客户预期管理已经做了，15 天到也不算差体验。相反，圣保罗预估 3 天实际 4 天到，虽然只晚了 1 天，但客户愤怒程度可能更高。**超时天数（delay magnitude）有时比超时率（delay rate）更有信息量。**

#### 5.10.2 评分分析

**分析前思考：**

1. **评分分布的"J 型曲线"是电商常态。** 大部分评分集中在 4-5 分，少数 1-2 分。不要对"均值 4.2"感到乐观——1 分和 5 分的数学均值是 3 分，但业务上 1 分的客户和 5 分的客户是完全不同的群体。看各分值的占比比看均值更有意义。
2. **评分和物流、价格、品类的交叉分析是整个项目最有价值的洞察。** "配送慢 → 评分低"验证了物流投资的价值（不只是快，是让客户开心→复购）。"贵的商品评分更低"可能是因为期望值更高。这些交叉结论是面试时展示"分析深度"的核心素材。
3. **差评集中在哪些品类？** 如果一个品类评分显著低于其他（比如电子产品 3.5 vs 全站 4.2），问题很可能不在平台而在供应链——品控、包装、运输损耗。这是平台可以干预的。

#### 5.10.3 支付行为分析

**分析前思考：**

1. **boleto 是巴西特色，这是面试官感兴趣的点。** boleto 是线下票据支付——客户在线下完订单，打印一张票据，去银行或便利店付现金。它代表了"无银行账户/无信用卡"人群。分析 boleto 用户的画像（客单价、品类偏好、地区分布）就是在分析"被金融体系排除在外的消费者如何参与电商"。
2. **分期行为的深层含义。** 信用卡分期在巴西非常普遍（不是"买不起"，而是惯常支付方式）。分期数高 ≠ 客户经济压力大，但分期数 + 品类交叉可以回答："买什么品类的人更倾向于分期？" 如果是家具/家电 → 大额消费分期合理。如果是食品/日用品 → 可能是经济压力信号。
3. **支付方式 × 州的差异。** 发达州（SP）信用卡占比高，北部偏远州 boleto 占比高 → 反映的是金融基础设施的差异，而不是支付偏好的差异。

#### 5.10.4 卖家绩效分析

**分析前思考：**

1. **卖家绩效分析回答的是"平台供给侧健康吗"。** 如果 Top 10 卖家占 80% GMV → 平台高度依赖少数卖家，任何一个离开都会伤筋动骨。如果卖家总数为 3095 但 80% 的卖家月均 < 1 单 → 供给侧活跃度不够，大量卖家没有真正参与。
2. **卖家评分 × 卖家规模。** 大卖家的评分比小卖家高还是低？大卖家通常物流更快（有仓储优势），但客服可能更机械。小卖家更在意每个客户，但配送能力弱。评分差异的根源才能指导平台的管理策略——是给大卖家设更高的服务标准，还是给小卖家提供物流支持？
3. **卖家地理集中度。** 如果 70% 的卖家集中在 SP → 平台上大部分商品从圣保罗发出，偏远地区的配送时效天然吃亏。这解释了物流分析中的"北部配送慢"——不是物流公司不行，是货源就在 2000 公里外。

#### 分析前思考

1. **这一步不是"写文档"，是"整理思维"。** 如果你在前面 8 天每个模块都问了"这个分析回答什么问题"，写结论就是把这些问题的答案串起来。如果某个模块你只有图和代码、说不出发现了什么 → 那个模块你做了但没理解。
2. **结论要有分层：事实 → 洞察 → 行动。**
   - 事实：SP 州占 GMV 的 xx%
   - 洞察：市场高度集中，仓储和物流 SP 最成熟，但过度依赖单一区域有风险
   - 行动：在 RJ/MG 建立二级仓储节点，降低跨州配送成本
   - 三层缺一不可。只有事实是"数据提取"，只有行动是"拍脑袋"。事实 + 洞察 + 行动才是"分析师的价值"。
3. **限制条件要主动说，不要藏着。** "只分析了 delivered 订单""Cohort 用 30 天近似月度""RFM 阈值是主观设定的"——这些主动写在结论后面。坦诚限制条件不是示弱，而是专业。面试官看到你承认限制，会认为你理解分析的边界，比"我分析得很完美"的印象好得多。
4. **结论要让非技术人员看懂。** 如果写出来一个完全不懂数据的人也能理解，那才是真正的"理解"。"首月留存率 45%" 不如 "每拉 100 个新客户，1 个月后只有 45 个还在买"。

```python
@st.cache_data
def load_data():
    """加载 9 张表 → JOIN → 清洗 → 返回分析用宽表"""
    # 读取所有表
    orders = pd.read_csv('data/olist_orders_dataset.csv')
    customers = pd.read_csv('data/olist_customers_dataset.csv')
    order_items = pd.read_csv('data/olist_order_items_dataset.csv')
    order_payments = pd.read_csv('data/olist_order_payments_dataset.csv')
    order_reviews = pd.read_csv('data/olist_order_reviews_dataset.csv')
    products = pd.read_csv('data/olist_products_dataset.csv')
    sellers = pd.read_csv('data/olist_sellers_dataset.csv')
    category_translation = pd.read_csv('data/product_category_name_translation.csv')

    # 日期转换
    date_cols = ['order_purchase_timestamp', 'order_approved_at',
                 'order_delivered_carrier_date', 'order_delivered_customer_date',
                 'order_estimated_delivery_date']
    for col in date_cols:
        orders[col] = pd.to_datetime(orders[col])

    # 只保留 delivered
    orders = orders[orders['order_status'] == 'delivered'].copy()

    # payments 聚合
    payments_agg = order_payments.groupby('order_id').agg({
        'payment_value': 'sum',
        'payment_type': lambda x: x.mode()[0] if not x.mode().empty else x.iloc[0],
        'payment_installments': 'max'
    }).reset_index()
    payments_agg.columns = ['order_id', 'total_payment', 'payment_type', 'max_installments']

    # reviews 聚合
    reviews_agg = order_reviews.groupby('order_id').agg({
        'review_score': 'min',
        'review_comment_message': 'first'
    }).reset_index()

    # JOIN
    df = order_items.merge(orders, on='order_id', how='left')
    df = df.merge(products[['product_id', 'product_category_name']], on='product_id', how='left')
    df = df.merge(category_translation, on='product_category_name', how='left')
    df = df.merge(customers[['customer_id', 'customer_unique_id', 'customer_state']],
                  on='customer_id', how='left')
    df = df.merge(payments_agg, on='order_id', how='left')
    df = df.merge(reviews_agg, on='order_id', how='left')
    df = df.merge(sellers[['seller_id', 'seller_state']], on='seller_id', how='left')

    # 计算列
    df['delivery_days'] = (df['order_delivered_customer_date'] -
                            df['order_purchase_timestamp']).dt.days
    df['item_total'] = df['price'] + df['freight_value']
    df['purchase_month'] = df['order_purchase_timestamp'].dt.to_period('M').astype(str)

    # 清洗
    df['product_category_name_english'] = df['product_category_name_english'].fillna('unknown')
    df = df[df['price'] > 0]
    df = df[df['delivery_days'] >= 0]

    return df
```

## 10. 面试可能问到的问题

### 10.1 Olist 特有的问题

**Q1：Olist 有 9 张表，你怎么把它们整合到一起的？遇到了什么问题？**
> 以 order_items 作为事实表（因为一个订单可以包含多个商品，事实表粒度是订单-商品级），通过 merge 逐步 JOIN 其他维度表。过程中注意三个问题：
> 1. order_payments 有多行对应一个 order_id（多次支付），需要先 groupby 聚合到订单级别
> 2. order_reviews 可能有重复 order_id，取了最小评分（保守估计）
> 3. 只保留了 order_status='delivered' 的订单，因为未完成的订单无法追踪物流时效和评分

**Q2：为什么只分析 delivered 订单？会不会有幸存者偏差？**
> 两个原因：1）物流时效分析要求订单必须有送达日期，canceled/unavailable 订单没有这个字段；2）评分也只在订单完成后产生。但我承认会存在偏差——比如被取消的订单背后可能是糟糕的购物体验，而它们不在分析范围内。更好的做法是单独分析取消订单的原因。

**Q3：customer_id 和 customer_unique_id 有什么区别？为什么用后者做 RFM？**
> customer_id 是每次下单时生成的 ID（同一个自然人在不同时间下单可能有不同的 customer_id）。customer_unique_id 才是真正的用户唯一标识。如果不用 unique_id，会把一个忠实客户拆成两个普通客户，R/F/M 全部失真。

**Q4：巴西特有的 boleto 支付是什么？你的分析对业务有什么价值？**
> Boleto 是巴西特有的线下票据支付方式（类似去银行/便利店扫码付现金）。分析发现 boleto 用户的分期率和客单价与信用卡用户不同，说明面向无信用卡人群的支付方式有独特的用户画像，可以针对性设计营销策略。

### 10.2 通用问题（与 UK 一样，调整了 Olist 上下文）

**Q5：RFM 的 qcut 分位数是什么概念？为什么用 5 档？**
> qcut 按分位数等分数据，5 档是二分位数的层级深化（业界惯例）。每个维度的 5 分制组合出 125 种 RFM Score，然后按规则合并成 8-10 个业务群体。

**Q6：Support / Confidence / Lift 怎么算的？各自代表什么？**
> Support（支持度）= 同时包含A和B的交易比例；Confidence（置信度）= 买了A的人中也买了B的比例；Lift（提升度）= Confidence / P(B)，衡量A对B的购买提升效果。Lift > 1 才有推荐价值。

**Q7：Cohort 分析的 PeriodNumber 怎么算的？为什么用 30 天近似？**
> (当前交易月份 - 客户首次购买月份) 的天数差 // 30。用 30 天近似一个月是行业习惯，精确到日历月的话需要考虑不同月份天数差异，而且代码更复杂。对宏观趋势分析来说，30 天近似已经足够。

**Q8：这个项目最大的难点是什么？**
> 准备你自己的答案。建议说："多表 JOIN 的数据整合——不是代码多难，而是需要理解每张表之间的关系，想清楚哪些表需要先聚合再 JOIN，否则行数会异常膨胀。"

**Q9：如果重新做这个项目，你会怎么改进？**
> 1）加入订单取消原因分析（目前只看了 delivered 订单）。2）对流式数据做监测（SKlearn 模型预测客户流失概率）。3）用 Folium 做巴西地图可视化。4）加入 A/B 测试能力。
